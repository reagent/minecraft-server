# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Purpose

Ansible-based toolkit to provision an on-demand Minecraft server on DigitalOcean. World data lives on a detachable DigitalOcean Block Storage volume so the droplet can be destroyed between sessions while preserving game state. Optional DNSimple integration manages an A record pointing at the droplet.

## Runtime requirements

- Python 3.11+ (pinned via `.python-version`; use `pyenv`). The managed host's Python interpreter is auto-detected by Ansible (no explicit `ansible_python_interpreter` is set); modern Ubuntu images ship Python 3 preinstalled.
- `ansible-core ~=2.18` pinned in `Pipfile`. DigitalOcean, DNSimple, and posix modules live in the `community.digitalocean`, `community.general`, and `ansible.posix` collections respectively. Collections are declared in `requirements.yml` and installed by `make setup` via `ansible-galaxy install -r requirements.yml`. The Minecraft server itself is configured by a small local role at `roles/minecraft/` (see `ansible.cfg` `roles_path=roles`) ‚Äî there are no external Galaxy roles to install.

## Common commands

All targets go through `pipenv` so the pinned Ansible is used:

```
make setup       # copies var templates, installs pip/pipenv deps, ansible-galaxy role install
make provision   # pipenv run ansible-playbook playbooks/create.yml
make destroy     # pipenv run ansible-playbook playbooks/destroy.yml
make init        # just materialize vars/credentials.yml and vars/server.yml from .templates/
```

`make setup` is idempotent only for the parts it owns. If `vars/credentials.yml` or `vars/server.yml` already exist, they are left alone (Make file-target rules). `keys/ansible` / `keys/ansible.pub` are generated on first `make provision` via `ssh-keygen`.

Running playbooks directly (bypasses pipenv ‚Äî will fail unless the correct Ansible is on PATH):

```
pipenv run ansible-playbook playbooks/create.yml
pipenv run ansible-playbook playbooks/destroy.yml
```

There are no tests, linters, or build steps. This repo is pure Ansible config.

## Required configuration before `make provision`

1. `vars/credentials.yml` ‚Äî must set `credentials.digital_ocean.api_key`. `create.yml` asserts this at the start. `credentials.dnsimple.{account_email,account_api_token}` is optional; DNS tasks run only when both are present (see conditional blocks in `playbooks/create.yml` and `playbooks/destroy.yml`).
2. `vars/server.yml` ‚Äî droplet sizing, region, image, volume name, and optional `dns:` block. `minecraft_home`, `minecraft_version`, and `minecraft_accept_eula` in this file are consumed by the local `minecraft` role, not by the playbooks themselves. `minecraft_accept_eula: true` is required ‚Äî the role asserts it and links to https://www.minecraft.net/en-us/eula on failure.
3. Ignored by git: `inventory`, `vars/*`. Do not commit credentials or the generated inventory.

## Architecture

Two playbooks, one module, one role ‚Äî the flow matters more than the file list.

### `playbooks/create.yml` ‚Äî two plays in sequence

1. **`hosts: localhost`** ‚Äî calls DigitalOcean API to upload the SSH key, create the droplet, create the block storage volume, attach volume to droplet. Writes `inventory` from `templates/inventory.j2` AND registers the same host in-memory via `add_host` so subsequent plays can target `minecraft` without a reload. Optionally upserts a DNSimple A record. Finally waits on port 22.
2. **`hosts: minecraft`** ‚Äî partitions the attached volume (`/dev/disk/by-id/scsi-0DO_Volume_<volume.name>`), makes ext4, mounts at `dirname(minecraft_home)` (i.e. the parent of `/srv/minecraft` ‚Üí `/srv`), then applies the local `minecraft` role which installs/configures the Minecraft server on top of that mount.

The mount point being `dirname(minecraft_home)` rather than `minecraft_home` itself is intentional: the role creates `minecraft_home` as a subdirectory of the mounted filesystem.

### `roles/minecraft/`

Inline role that replaces the unmaintained `devops-coop.minecraft` (#13). Installs `openjdk-21-jre-headless`, creates a system `minecraft` user/group, resolves the `server.jar` URL + sha1 from the Mojang version manifest (`https://launchermeta.mojang.com/mc/game/version_manifest_v2.json`) for `{{ minecraft_version }}`, downloads with checksum verification, writes `eula.txt`, and installs a systemd unit (`/etc/systemd/system/minecraft.service`). The unit name `minecraft` matches what `playbooks/destroy.yml` stops before unmounting the volume.

### `playbooks/destroy.yml`

Stops the `minecraft` service, unmounts the volume, destroys the droplet (`delegate_to: localhost`), deletes the local `inventory` file, and removes the DNSimple record if configured. The block storage volume is deliberately **not** deleted ‚Äî world data persists for the next `make provision`.

### `ansible.cfg`

- `inventory=inventory` ‚Äî the file generated by the first play
- `private_key_file=keys/ansible` ‚Äî the key generated by `make provision`
- `roles_path=roles` ‚Äî only the local `roles/minecraft/` role is used; no Galaxy roles

## Gotchas

- The default template targets Ubuntu 22.04 LTS (`image: ubuntu-22-04-x64`). 24.04 also works; bump when 22.04 nears its April 2027 standard-support end. DO periodically deprecates old slugs ‚Äî see https://slugs.do-api.dev/ for current options.
- The `community.digitalocean` collection is marked deprecated upstream and will be removed from Ansible 13. No drop-in replacement yet ‚Äî keep an eye on `digitalocean.cloud` (official vendor collection) as a future migration path.
- `make setup` runs `pip install -U -r requirements.txt` against the ambient Python, which installs/upgrades `pipenv` (pinned to `>=2024.0`) globally. Downgrade risk of the previous `==10.1.2` pin is gone; the remaining rough edge is that it's a global install rather than an isolated tool install via pipx. #14 tracks moving off pipenv entirely.
- Relative paths in playbooks (`../vars/...`, `../keys/...`, `../templates/...`, `../inventory`) assume playbooks are run from the repo root, which is what `make` does. Running `ansible-playbook` from inside `playbooks/` will not work.
