# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Purpose

Ansible-based toolkit to provision an on-demand Minecraft server on DigitalOcean. World data lives on a detachable DigitalOcean Block Storage volume so the droplet can be destroyed between sessions while preserving game state. Optional DNSimple integration manages an A record pointing at the droplet.

## Runtime requirements

- Python 3.11+ (pinned via `.python-version`; use `pyenv`). The managed host's Python interpreter is auto-detected by Ansible (no explicit `ansible_python_interpreter` is set); modern Ubuntu images ship Python 3 preinstalled.
- `ansible-core ~=2.18` pinned in `Pipfile`. DigitalOcean and DNSimple modules live in the `community.digitalocean` and `community.general` collections respectively — pinned in `collections/requirements.yml` and installed into the user collections path via `ansible-galaxy collection install` (invoked from `make setup`).
- `devops-coop.minecraft` role pinned to `v3.1.0` in `requirements.yml`. Role is installed into `vendor/roles/` (see `ansible.cfg` `roles_path`).

## Common commands

All targets go through `pipenv` so the pinned Ansible is used:

```
make setup       # copies var templates, installs pip/pipenv deps, ansible-galaxy role install
make provision   # pipenv run ansible-playbook playbooks/create.yml
make destroy     # pipenv run ansible-playbook playbooks/destroy.yml
make init        # just materialize vars/credentials.yml and vars/server.yml from .templates/
```

`make setup` is idempotent only for the parts it owns. If `vars/credentials.yml` or `vars/server.yml` already exist, they are left alone (Make file-target rules). `keys/ansible` / `keys/ansible.pub` are generated on first `make provision` via `ssh-keygen`.

Running playbooks directly (bypasses pipenv — will fail unless the correct Ansible is on PATH):

```
pipenv run ansible-playbook playbooks/create.yml
pipenv run ansible-playbook playbooks/destroy.yml
```

There are no tests, linters, or build steps. This repo is pure Ansible config.

## Required configuration before `make provision`

1. `vars/credentials.yml` — must set `credentials.digital_ocean.api_key`. `create.yml` asserts this at the start. `credentials.dnsimple.{account_email,account_api_token}` is optional; DNS tasks run only when both are present (see conditional blocks in `playbooks/create.yml` and `playbooks/destroy.yml`).
2. `vars/server.yml` — droplet sizing, region, image, volume name, and optional `dns:` block. `minecraft_home` and `minecraft_version` in this file are consumed by the `devops-coop.minecraft` role, not by the playbooks themselves.
3. Ignored by git: `inventory`, `vars/*`. Do not commit credentials or the generated inventory.

## Architecture

Two playbooks, one module, one role — the flow matters more than the file list.

### `playbooks/create.yml` — two plays in sequence

1. **`hosts: localhost`** — calls DigitalOcean API to upload the SSH key, create the droplet, create the block storage volume, attach volume to droplet. Writes `inventory` from `templates/inventory.j2` AND registers the same host in-memory via `add_host` so subsequent plays can target `minecraft` without a reload. Optionally upserts a DNSimple A record. Finally waits on port 22.
2. **`hosts: minecraft`** — partitions the attached volume (`/dev/disk/by-id/scsi-0DO_Volume_<volume.name>`), makes ext4, mounts at `dirname(minecraft_home)` (i.e. the parent of `/srv/minecraft` → `/srv`), then applies the `devops-coop.minecraft` role which installs/configures the Minecraft server on top of that mount.

The mount point being `dirname(minecraft_home)` rather than `minecraft_home` itself is intentional: the role creates `minecraft_home` as a subdirectory of the mounted filesystem.

### `playbooks/destroy.yml`

Stops the `minecraft` service, unmounts the volume, destroys the droplet (`delegate_to: localhost`), deletes the local `inventory` file, and removes the DNSimple record if configured. The block storage volume is deliberately **not** deleted — world data persists for the next `make provision`.

### `ansible.cfg`

- `inventory=inventory` — the file generated by the first play
- `private_key_file=keys/ansible` — the key generated by `make provision`
- `roles_path=vendor/roles:roles` — `ansible-galaxy install` writes the pinned role into `vendor/roles/`

## Gotchas

- The playbooks target Ubuntu 16.04 (`image: ubuntu-16-04-x64` in `.templates/server.yml`). DigitalOcean has since deprecated that image slug; if provisioning fails with an image-not-found error, update `vars/server.yml` to a supported slug rather than patching the playbook.
- The `community.digitalocean` collection is marked deprecated upstream and will be removed from Ansible 13. No drop-in replacement yet — keep an eye on `digitalocean.cloud` (official vendor collection) as a future migration path.
- `make setup` runs `pip install -U -r requirements.txt` against the ambient Python, which installs `pipenv==10.1.2` globally. If a newer pipenv is already installed system-wide, this can downgrade it.
- Relative paths in playbooks (`../vars/...`, `../keys/...`, `../templates/...`, `../inventory`) assume playbooks are run from the repo root, which is what `make` does. Running `ansible-playbook` from inside `playbooks/` will not work.
